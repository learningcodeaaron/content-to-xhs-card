# Token 使用优化

## 读什么

普通文章流程只读：

1. `analysis.compact.json`
2. 用户确认的 `拆分方案.md`
3. 需要复用时的 `卡片数据.json`

不要读取模板源码、生成后的完整 HTML、Base64 图片或已经物化的全部元素，除非任务就是开发编辑器。

## 谁负责什么

- 脚本：清洗、结构提取、伪标题、表格和列表识别、卡数预算、布局候选、迁移、校验、HTML 打包。
- 模型：中心命题、语义冲突、完整文案压缩、作者原意核对、最终叙事。
- 编辑器：模板物化、自由移动、素材替换、持久化与 PNG。

模型优先返回字段 patch，而不是重写整份 deck。置信度高的脚本字段不重复生成。

## 预算

- `analyze_article.mjs` 默认把紧凑 JSON 控制在约 4,800 字符以内，并保留最多 16 条分类记录。
- 确认前只生成 Markdown；确认后由 `plan_to_deck.mjs` 转 JSON，避免模型重复转录。
- 已有 v2 JSON 时不再读取原文。
- 对第二篇股民画像样本，脚本可把 12,878 字符原文清理为约 5,360 字符语义正文，并生成约 4,550 字符的紧凑模型输入。
